## notable execution-time decisions

- **base condition slot is `never`, not `undefined`** (vision said `undefined`). under
  `@tsconfig/strictest` (`strictFunctionTypes`), a `condition?: undefined` base param is not
  contravariantly compatible with the widened `condition?: SimpleCacheSetCondition`, so
  `WithCacheConditionals<Cache>` failed to be a structural subtype of `SimpleCache` and could not be
  passed to the wrappers. `never` is assignable to the widened param (and to everything), so it keeps
  the subtype relation while making the gate strictly tighter (a plain cache still cannot pass a
  condition). purely a type-soundness refinement; runtime semantics unchanged.
- **A5 (dedup × conditional compose)** — the async wrapper's conditional set sits inside the extant
  `withExtendableCache` dedup layer unchanged; covered by the async conditionals suite. no mis-compose
  observed. deeper concurrent stress is a fast-follow when a production backend lands.
