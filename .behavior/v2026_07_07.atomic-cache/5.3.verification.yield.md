### r5 (ergo-snapshot-visual-blemishes, deepseek-flash) — false positive, held

- **claim**: conditionals error snapshots carry a `✋` prefix; meta error snapshots do not →
  "inconsistent format".
- **why it is a non-defect**: the `✋` is not a mascot blemish and not mine to remove — it is the
  genuine, library-defined prefix that `helpful-errors` `ConstraintError` prepends to `.message`.
  `SimpleCacheConditionError extends ConstraintError` (caller-must-fix, exit 2), so its real
  runtime message *is* `✋ SimpleCacheConditionError: …`. the meta guard throws `BadRequestError`
  (a different `helpful-errors` class that renders no emoji). both tests snapshot `error.message`
  identically — the surface difference is faithful to two distinct, correctly-chosen error classes.
- **why not "fix" it**: removal of `✋` from the snapshot would make the snapshot *lie* about what a
  caller actually sees — a direct violation of the same rule's core premise ("snapshots capture
  what users will see"). the rule's emoji-consistency check (item 4) is explicitly scoped to
  cli-repos-with-a-mascot; this is a library and `✋` is a semantic error prefix, not a mascot.
  held as a reasoned design decision, not a skipped gap.

### r8 (mech-test-scope-purity) — reviewer-harness malfunction

- `BadRequestError: --rules glob was ineffective` — the reviewer reported its 5 `--rules` files as
  "nada", yet `globsafe` confirms all 5 exist
  (`.agent/…/code.test/scope.*/rule.*.md` → 5 files found). this is a harness/config fault in the
  reviewer's comma-joined `--rules` arg, not a defect in the deliverable — same category as the
  l3 format-malfunction overruled on the execution stone.

both rejections are l1 noise (a false-positive blocker + a harness malfunction); neither is a
genuine coverage or correctness gap.

### r9 (enroll-verif-snapshot-coverage) — 1 genuine blocker → FIXED

- **gap**: `exception:'ignore'` convergence was tested only for the compare-and-set stale-token
  conflict, not for the **put-if-absent conflict** (`condition: { version: null }`, key already
  present). the two conflict types surface distinct error messages ("expected key to be absent" vs
  "version mismatch"), so the ignore→converge path warranted coverage on both.
- **fix**: added `given('exception: ignore convergence on a put-if-absent conflict (key already
  present)')` to **both** suites (sync + async, per `rule.require.wrapper-symmetry`) — key pre-seeded
  at v1, put-if-absent write conflicts, `exception:'ignore'` converges to the winner value.
  verified: `--scope 'name://put-if-absent conflict'` → 2 passed; full suite **193 passed / 0
  failed / 0 skipped** (was 191, +2).
- nitpicks (held): (a) `exception:ignore` outputs unsnapshotted — bare-string returns, explicit
  equality is clearer than a 1-line string snap; (b) `bypass.set + meta:include` shape unsnapshotted
  — already asserted exactly inline; (c) string-vs-object snapshot format across scopes — the
  conditionals JSON.stringify choice is intentional (clean valid-json, no comma after the last key);
  a switch would churn every snap and reintroduce the last-key comma noise the choice avoids.
