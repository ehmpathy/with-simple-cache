# vision — atomic-cache (conditional writes)

## .what (one-liner)

give `with-simple-cache` **atomic conditional writes** — a cache advertises "i support
compare-and-set", the wrappers gate their write on a version precondition
(`{ condition: { version: null | string } }`), and consumers read the version back via
`meta:'include'`. `version: null` = "write only if absent"; `version: '<token>'` =
"write only if current version === token".

this mirrors the extant `WithCacheUri` / `HasCacheUri` / `meta:'include'` pattern: the
package **defines the vocabulary and gates on the capability**; each backend **supplies
the impl**. per vlad, this wish ships the **full stack** — vocabulary + version-in-meta
+ the `condition` wrapper option (not descriptor-only).

---

## the outcome world

### day-in-the-life

today, `SimpleCache.set(key, value, { expiration })` is **last-writer-wins**. two
processes that both miss the cache both compute and both blindly overwrite. there is no
way to say "only write if nobody beat me to it" or "only write if the value hasn't
changed since i read it."

after this wish, a cache can advertise conditional writes, and a consumer can lean on
them for **atomicity**:

- **put-if-absent** — first writer wins; every other either surfaces a conflict
  (throw, default) or converges on the winner's value (`exception: 'ignore'`)
  (compute-once across processes, distributed-lock acquire, stampede protection)
- **compare-and-set** — "update only if the version i read is still current"
  (optimistic concurrency; safe read-modify-write)

### before / after

```ts
// before — blind overwrite, races clobber each other
await cache.set(key, next, { expiration });

// after — same set(), one more option next to expiration
await cache.set(key, next, {
  expiration,
  condition: { version: null }, // null = "must still be absent"; or a token = "must match"
});
// ^ throws SimpleCacheConditionError if the precondition is not met
```

`condition` is **just another option on `set`**, exactly like `expiration` — present or
absent. no new methods. the version to match comes from `meta:'include'` (below).

### the "aha" moment

> "the cache contract already knows how to say *where* a value lives (`uri`). now it
> can say *whether my write won the race* — and it's the **same capability-descriptor
> shape**, so all my backends (in-memory CAS, disk `O_EXCL`, s3 `If-None-Match`)
> declare conformance to one canonical contract."

atomicity stops being a per-backend snowflake and becomes a **typed capability** that
consumers require uniformly.

### the bridge — `meta:'include'` surfaces the version

the wrapper already exposes cache metadata via `meta:'include'` (today: `cached.uri`).
when the cache **also** supports conditionals, the wrapped call should surface the
**version** in the very same envelope — so a caller reads it off the result and hands
it straight back as the `condition` on a later `set`:

```ts
const getRecord = withSimpleCacheAsync(fetchRecord, { cache, meta: 'include' });

const { output, cached } = await getRecord({ id: 'abc' });
// cached.uri     — present because cache HasCacheUri
// cached.version — present because cache HasCacheConditionals

// later, condition the next write on what we read
await cache.set(key, next, { condition: { version: cached.version } });
```

this version-in-meta path is the **read side** — it *exposes* the version so callers can
condition. the **write side** is the `condition` option on `set` (and on the wrapper).
together they close the loop: read a version off `meta:'include'`, then pass it as
`condition` on the next `set`. no bespoke read/write methods — just `meta` + a `set`
option.

### contract inputs & outputs

**`condition` is an option on BOTH `get` and `set`** (composable, uniform), plus a
`version(key)` read-surface. its type is **dependent on the cache shape**: `undefined`
(unusable) unless the cache `HasCacheConditionals`. get/set **return shapes are unchanged**
— minimal downstream churn.

**(1) make the get/set options generic on the condition type.** base `get`/`set` carry
`condition?: undefined` — present in the signature, but only omission/`undefined` compiles
until a capability widens it:

```ts
/** the shape of a version precondition — usable on BOTH get and set */
export type SimpleCacheSetCondition = { version: string | null };
// version: null = "must be absent" (put-if-absent); '<token>' = "must match" (compare-and-set)

/** get options; condition slot is `undefined` until a cache supplies its type */
export interface SimpleCacheGetOptions<TCondition = undefined> {
  condition?: TCondition;
}
/** set options; condition ordered AFTER expiration (per vlad) */
export interface SimpleCacheSetOptions<TCondition = undefined> {
  expiration?: IsoDuration | null;
  condition?: TCondition; // ← after expiration; base: undefined; conditional: SimpleCacheSetCondition
}

export interface SimpleCacheSync<T> {
  get: (key: string, options?: SimpleCacheGetOptions) => T | undefined;
  set: (key: string, value: T | undefined, options?: SimpleCacheSetOptions) => void;
  //                condition?: undefined on both ^
}
// (SimpleCacheAsync identical, Promise-wrapped)
```

so on a **plain** cache, `cache.get(k, { condition: { version } })` / `cache.set(...)` is a
**compile error** (`{ version }` is not assignable to `undefined`). exactly vlad's ask:
`condition` must be `undefined` unless the cache `HasCacheConditionals`.

**(2) `WithCacheConditionals` adds `version(key)` and widens `condition` on `get` + `set`:**

```ts
export type WithCacheConditionals<TCache extends SimpleCache<any>> =
  Omit<TCache, 'get' | 'set'> & {
    // read the current version token (undefined if absent) — mirror of uri(key)
    version: TCache extends SimpleCacheAsync<any>
      ? (key: string) => Promise<string | undefined>
      : (key: string) => string | undefined;

    // get, now with an optional version condition that validates the read
    // (throws SimpleCacheConditionError if the version no longer matches)
    get: TCache extends SimpleCacheAsync<infer T>
      ? (key: string, options?: SimpleCacheGetOptions<SimpleCacheSetCondition>) => Promise<T | undefined>
      : TCache extends SimpleCacheSync<infer T>
      ? (key: string, options?: SimpleCacheGetOptions<SimpleCacheSetCondition>) => T | undefined
      : never;

    // set, now with an optional version condition that gates the write (atomic CAS)
    set: TCache extends SimpleCacheAsync<infer T>
      ? (key: string, value: T | undefined, options?: SimpleCacheSetOptions<SimpleCacheSetCondition>) => Promise<void>
      : TCache extends SimpleCacheSync<infer T>
      ? (key: string, value: T | undefined, options?: SimpleCacheSetOptions<SimpleCacheSetCondition>) => void
      : never;
  };

/** detect conditional capability off the distinctive `version()` method (mirror of HasCacheUri) */
export type HasCacheConditionals<C> = C extends { version: (key: string) => any }
  ? true
  : false;
```

**(3) a typed error consumers branch on:**

```ts
import { ConstraintError } from 'helpful-errors';

/**
 * thrown by get(...) or set(...) when a given condition.version precondition is not met
 * (the key was present/absent contrary to expectation, or the version did not match)
 *
 * extends ConstraintError — a caller-must-fix constraint violation (exit 2)
 */
export class SimpleCacheConditionError extends ConstraintError { /* + metadata: key, condition, found */ }
```

> **the crux (per vlad)**: the type system alone decides whether `condition` is passable —
> via the `SimpleCacheGetOptions<TCondition>` / `SimpleCacheSetOptions<TCondition>` generics. base cache
> → `condition?: undefined` (rejected); `WithCacheConditionals` → `condition?:
> SimpleCacheSetCondition` (accepted). same gate technique as `meta:'include'` on `HasCacheUri`.
>
> **atomicity** (see the crystal timeline below): `version(key)` then `get(key, { condition:
> { version } })` couples value to version by *validation* — a torn read throws + retries, so
> no `(value, version)` desync. the earlier (wrong) sketch used `setConditional` /
> `getWithVersion` methods and a decoupled `version()` read — dropped. `condition` rides on
> get/set; version rides on `version()` + the validation get.

secondary deliverable = **`meta:'include'` surfaces the version** (per vlad). the
`cached` envelope becomes a capability-gated composite, and `meta:'include'` becomes
available when the cache has **uri OR conditionals** (today: uri only):

```ts
/**
 * the metadata surfaced alongside output when meta:'include'
 * — each field appears only when the cache advertises that capability
 */
type CacheMetaOutput<C extends SimpleCache<any>> =
  & (HasCacheUri<C>          extends true ? { uri: string }     : {})
  & (HasCacheConditionals<C> extends true ? { version: string } : {});

// meta:'include' allowed when the cache has *something* worth surfacing
type CacheMetaChoice<C extends SimpleCache<any>> =
  HasCacheUri<C> extends true
    ? 'include' | 'exclude'
    : HasCacheConditionals<C> extends true
    ? 'include' | 'exclude'
    : 'exclude';
```

### atomicity — how `version()` + a validating `get` guarantee it (crystal timeline)

it was not obvious that a separate `version()` method can be atomic. it can — because the
`condition` on `get`/`set` **validates** the version at the backend, in one indivisible op.
here are three timelines with a reader `R` and a concurrent writer `W`. `Vn` = the key's
version at each moment.

**❌ scenario A — the desync bug (a *decoupled* read, which we do NOT do):**

```
t0  R: get(key)                                   → value A     (key at V1)
t1  W: set(key, B)                                → key now V2
t2  R: version(key)                               → V2          ← DESYNC: A is V1's value, tagged V2
t3  R: set(key, f(A), { condition:{ version:V2 }})→ current is V2 → CAS SUCCEEDS
        └ 💥 lost update: B is clobbered by f(A), which never saw B
```

the trap: reading value and version as two *unvalidated* steps lets them drift apart.

**✅ scenario B — validating read, happy path (what we do):**

```
t0  R: version(key)                               → V1
t1  R: get(key, { condition:{ version:V1 }})      → value A     (still V1) ✓ (A,V1) provably coupled
t2  R: compute next = f(A)
t3  R: set(key, next, { condition:{ version:V1 }})→ still V1 → write, key → V2  ✓ atomic update
```

`R` only ever holds a `(value, version)` pair the backend **confirmed** matched.

**✅ scenario C — validating read, race detected (safety kicks in):**

```
t0  R: version(key)                               → V1
t1  W: set(key, B)                                → key now V2
t2  R: get(key, { condition:{ version:V1 }})      → V1 no longer current → THROWS SimpleCacheConditionError
        └ R retries: version(key)→V2, get(cond V2)→B ✓ (B,V2) coupled — then writes on V2
```

and if the race lands between `R`'s read and write instead:

```
t3  R: set(key, next, { condition:{ version:V1 }})→ key moved to V2 → THROWS → R retries from t0
```

**why it's atomic, in one line:** every `condition` is checked *inside* the backend's
single get/set op, so `R` can never (a) read a value that disagrees with its version, nor
(b) write over a change it did not observe. a torn interleaving is always *detected* and
*retried*, never silently lost. opaque, per-write-unique versions (A2) mean no ABA repeat.

> the separate `version()` is safe **because** the `get`/`set` `condition` re-checks it
> atomically — the token is a claim the backend validates, not a value you trust blind.

### what leveraging it looks like

```ts
// a consumer that *requires* the capability at compile time
const acquireLock = async (
  cache: WithCacheConditionals<SimpleCacheAsync<string>>, // plain SimpleCache rejected here
  key: string,
) => {
  try {
    await cache.set(key, 'held', {
      expiration: { seconds: 30 },
      condition: { version: null }, // must be absent
    });
    return { acquired: true };
  } catch (error) {
    if (error instanceof SimpleCacheConditionError) return { acquired: false };
    throw error;
  }
};
```

### timeline

1. read via wrapper `meta:'include'` → `{ output, cached: { version } }` (or `undefined`)
2. compute next value
3. write with the option: `set(key, next, { condition: { version: version ?? null } })`
4. on `SimpleCacheConditionError` → default throws to the caller; `exception: 'ignore'`
   swallows + re-reads the winner's value

## mental model

- **describe-to-a-friend**: "it's `If-Match` / `If-None-Match` headers, but for any
  cache — the cache tells you a version when you read, and you hand it back when you
  write so your write only lands if the value hasn't changed underneath you."
- **analogy**: HTTP conditional requests (ETag), database optimistic-locking
  (`WHERE version = ?`), s3 `If-None-Match: *`, filesystem `open(O_CREAT|O_EXCL)`,
  redis `SET NX` / `WATCH`+`MULTI`.
- **our terms vs their terms**:

  | concept | wish term | peer (#57) term | this vision |
  |---------|-----------|-----------------|-------------|
  | capability type | (n/a) | `WithCacheConditionals` | `WithCacheConditionals` (adds `version(key)`, mirror of `uri(key)`) |
  | detector | (n/a) | `HasCacheConditionals` | `HasCacheConditionals` (off `version()`) |
  | version read | (n/a) | `getWithVersion` method | **`version(key)`** (token only); surfaced to callers via `meta:'include'` → `cached.version` |
  | gated read | (n/a) | (n/a) | **`get(..., { condition: { version } })`** — validates the read |
  | gated write | `condition: { version }` | `setConditional(..., { ifVersion })` | **`set(..., { condition: { version } })`** — option on `set`, after `expiration` |
  | "must be absent" | `version: null` | `ifVersion: "absent"` | `version: null` ✅ (wish shape) |
  | conflict signal | (n/a) | `SimpleCacheConditionError` | `SimpleCacheConditionError` (thrown by `get` or `set`) |

## evaluation

### how well it solves the goal

- **strong** on the peer's framing: adds atomicity to the *contract vocabulary* where
  it belongs, composes cleanly with `WithCacheUri` (a cache can advertise both), and
  keeps the base `SimpleCache` honest (a plain cache still can't lie about atomicity).
- mirrors an **already-validated pattern** in this repo (`meta:'include'` gated on
  `HasCacheUri`), so the shape, tests, and symmetry rules are known-good.

### pros

- capability-gated types → a plain `SimpleCache` is **rejected at compile time** where
  conditionals are required (same as `meta:'include'`)
- backend-agnostic → in-memory, disk, s3 all conform to one shape
- additive & backwards-compatible → base `SimpleCache` unchanged; no extant caller
  breaks

### cons / costs

- **version is opaque** — consumers must treat it as an equality token, never parse or
  order it. easy to misuse.
- **the read-then-write pattern needs care** — a naive `get()` + `version()` (unvalidated)
  desyncs (see scenario A); the *validating* `get(..., { condition })` avoids it but callers
  (and the wrapper) must use that pattern, not the naive one.
- **no production backend ships here** — the wrappers consume the capability, but a
  real conditional-capable cache (in-repo test asset + sibling-repo backends) is needed
  before the value lands in production.
- **conflict response carries a mode flag** — `exception: 'throw' | 'ignore'` on the
  `condition` shape governs whether a precondition miss propagates or is swallowed +
  re-read. throw is the default for all preconditions; both modes must be covered by
  tests.

### edgecases & pit-of-success

| edgecase | how the contract keeps you safe |
|----------|-------------------------------|
| set-if-absent but key appeared | `set(..., { condition:{ version:null } })` throws `SimpleCacheConditionError` |
| read torn by a concurrent write | validating `get(..., { condition })` throws → caller retries (scenario C) — never a desync |
| compare-and-set but key vanished (expired) | throws — caller re-reads, sees absent, re-decides |
| consumer forgets to handle conflict | it's a **typed** error → easy to branch; unhandled = loud throw, not silent loss |
| plain cache passed where conditionals required | compile-time rejection via `HasCacheConditionals` gate |
| `value: undefined` (delete-if-version) | conditional delete *could* mirror extant `set(key, undefined)` — speculative, unverified per backend (see A4) |

## open questions & assumptions

### 🔑 scope — DECIDED: full wrapper-integration is mandatory

per vlad, the deliverable is **not** descriptor-only. all three tiers ship together:

- **tier 1 — vocabulary**: `SimpleCacheSetCondition` / `SimpleCacheSetOptions<TCondition>` /
  `WithCacheConditionals` / `HasCacheConditionals` / `SimpleCacheConditionError`, beside
  `SimpleCache`.
- **tier 2 — version-in-meta**: when the cache `HasCacheConditionals`, `meta:'include'`
  surfaces `cached.version`, and `meta:'include'` widens to be available for
  conditionals-capable caches (see `CacheMetaOutput` / `CacheMetaChoice` above).
- **tier 3 — the `condition` wrapper option (mandatory)**: `withSimpleCache` /
  `withSimpleCacheAsync` gain a `condition` option so the get→compute→**set** flow calls
  `cache.set(key, value, { condition })`, gated on `HasCacheConditionals` (a plain cache
  is rejected at compile time, as `meta:'include'` is gated on `HasCacheUri`).

the wrapper `condition` option — `version` is a **dynamic extractor over the full
operation input**, a mirror of the extant `cache` choice (`(...args) => cache`) and the
`bypass` predicates (`(...args) => boolean`):

```ts
// condition, present in the wrapper options ONLY when HasCacheConditionals<C>
condition?: {
  // static value OR an extractor over the operation's full args (input, context, ...)
  version: (string | null) | ((...args: Parameters<L>) => string | null);
  exception?: 'throw' | 'ignore';
};
```

usages:

```ts
const getRecord = withSimpleCacheAsync(fetchRecord, {
  cache,                                          // must satisfy HasCacheConditionals
  condition: { version: null },                   // put-if-absent; throw on conflict (default)
  // condition: { version: null, exception: 'ignore' },   // put-if-absent; swallow + converge
  // condition: { version: '<token>' },           // compare-and-set (static token)
  // condition: {                                  // ← version extracted from the full input:
  //   version: (input, context) => input.expectedVersion ?? null,
  // },
});
```

the `condition.version` extractor reuses the same call convention as `cache`
(`getCacheFromCacheChoice.ts:21` — `(...args: LI) => V`, invoked with `...forInput`).

- **`exception: 'throw'` (default, per vlad)** → on precondition miss the wrapper
  propagates `SimpleCacheConditionError`. applies to **both** `version: null` and a token.
- **`exception: 'ignore'`** → on precondition miss the wrapper **swallows** the error,
  **re-reads, and returns the winner's value** (per vlad — "gotta converge"; the loser's
  freshly-computed output is discarded), consistent with the extant get-after-set
  invariant at `withSimpleCache.ts:256-259`.

the wrapper calls `cache.set(key, value, { expiration, condition: { version } })` (the
`set` throws on miss); `exception` governs only whether the **wrapper** catches.

- `condition: { version: null }` → **cross-process compute-once** (only the first misser
  persists). upgrades today's *in-process* `withExtendableCache` dedup to a
  *cross-process* guarantee. pair with `exception: 'ignore'` to converge silently.
- `condition: { version: '<token>' }` → compare-and-set (optimistic concurrency).
- **symmetry**: the option is identical on both wrappers (`rule.require.wrapper-symmetry`).

### decided (per vlad)

- **precondition shape** = `{ condition: { version: null | string } }` (wish shape),
  `version: null` = "must be absent". we do **not** use the peer's `ifVersion` /
  `"absent"` vocabulary.
- **wrapper integration** = mandatory (tier 3 above).
- **conflict resolution** = `exception: 'throw' | 'ignore'` on the `condition` shape;
  **throw by default** for *both* `version: null` and a version token. `exception:
  'ignore'` swallows the conflict (compute-once escape hatch), re-reads, and returns the
  **winner's value** (per vlad — "gotta converge"; A1 decided).
- **static + derived `condition`** = **both supported** — a literal `{ version }` and an
  extraction fn `(input, context) => ({ version })` (mirrors the extant `cache` choice +
  `bypass` predicate style).
- **`condition` + `bypass` = independent/orthogonal** — `bypass.set` short-circuits
  *before* any set, so when it fires no conditional write happens at all. the two options
  do not interact.
- **test backend** = add an **in-repo `.test.assets` conditional cache** (a small CAS
  impl) to exercise tier 3 now. **fast-follow via radio tasks**: upgrade
  `simple-on-disk-cache` (#43) and `simple-in-memory-cache` to conform to
  `WithCacheConditionals` for production use.

### assumptions made

- version is an **opaque, equality-only** token (etag semantics), not monotonic/ordered.
- the wrappers **consume** the capability (tier 3), but the package still ships **no
  production backend** impl — an in-repo `.test.assets` conditional cache exercises it,
  and production backends conform via fast-follow radio tasks (`simple-on-disk-cache#43`,
  `simple-in-memory-cache`, `sdk-aws-s3#5`).

  name. new descriptors + error must be added there.
- **error convention in-repo** — `BadRequestError` is a hand-rolled `Error` subclass
  with a metadata blob (`src/utils/errors/BadRequestError.ts`). `SimpleCacheConditionError`
  should follow this shape (pending open-question #3).
- **the extant dedup is in-process only** — `withSimpleCacheAsync` uses
  `withExtendableCache` for in-memory promise dedup (`withSimpleCacheAsync.ts:314-340`).
  tier 3 (compute-once) extends this guarantee **across processes** via
  `set(..., { condition: { version: null } })`.

---

## what is awkward?

- **test backend is a hard prerequisite.** wrapper-integration is mandatory + integration
  tests forbid mocks → we cannot ship tier 3 without a **real conditional-capable cache**
  to exercise it. resolved by an in-repo `.test.assets` conditional cache now, with
  production backends via fast-follow radio tasks — but it means this wish's scope
  includes writing a small correct CAS impl, not just wiring options.
- **conflict handling is one explicit option, not a hidden fork.** the `exception:
  'throw' | 'ignore'` flag makes the compute-once-vs-surface choice explicit per call;
  throw is the default for all preconditions. the only subtlety: `exception: 'ignore'`
  re-reads and returns the *winner's* value, so the loser's freshly-computed output is
  discarded — correct for convergence, but worth documenting so callers aren't surprised
  their computed value was dropped.
- **`get` + `version()` is a two-call read for CAS.** to get a coupled `(value, version)`,
  a caller does `version()` then `get(..., { condition })` (or reads `cached.version` off
  `meta:'include'`). it is safe (the validating get prevents desync) but it *is* two ops —
  backends may internally memoize per `(key, version)` to dedupe. `version()` only appears
  on `WithCacheConditionals`, so it stays invisible unless you opted into the capability.
- **opaque-version discipline is unenforceable by types.** no rule stops a consumer
  from `parseInt(version)`. we rely on docs + the etag analogy to keep them in the pit
  of success.
- **`null`-as-absent overloads `null`.** `SimpleCache` already treats `undefined` as
  "not found" and `null` as a legitimate storable value in some backends; `condition:
  { version: null }` for "must be absent" is a *third* sense of nullish. we adopt it
  (it is the wish shape), but it is a real ergonomic edge to document clearly.
