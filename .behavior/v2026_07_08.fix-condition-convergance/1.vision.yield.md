# 1.vision — fix condition convergence across package boundaries

## the core defect (one sentence)

the conditional-write converge guard recognizes a `SimpleCacheConditionError` only when it is the *exact same constructor* the wrapper imported — so when a real backend cache (a separate package) throws its *own* `SimpleCacheConditionError`, `instanceof` fails and the wrapper re-throws instead of converging.

---

## the outcome world

### day-in-the-life

a caller wraps a real cache backend — say an s3-backed conditional cache from a *separate npm package* — and asks for put-if-absent convergence:

```ts
const runOnce = withSimpleCacheAsync(computeExpensiveThing, {
  cache: myS3ConditionalCache,          // a different package
  serialize: { key: (input) => input.id },
  condition: { version: null, exception: 'ignore' }, // "if someone beat me to it, converge on their value"
});

// two racers call at once; one wins the write, one loses
const result = await runOnce({ id: 'lock' });
// caller expects: the loser quietly converges on the winner's value
```

- **before**: the losing racer's `cache.set` throws the backend's `SimpleCacheConditionError`. the wrapper's guard does `error instanceof SimpleCacheConditionError` against *its own* class → `false` → the conflict propagates as an unhandled throw. `exception: 'ignore'` silently does none of its job. the caller gets a crash instead of convergence.
- **after**: the wrapper recognizes the backend's error structurally (`instanceof ConstraintError && error.constructor?.name === 'SimpleCacheConditionError'`) → converges per the exception mode → the loser re-reads and returns the winner's value.

### before/after contrast

| dimension | before | after |
|-----------|--------|-------|
| same-package error (in-repo tests) | converges ✅ | converges ✅ (unchanged) |
| cross-package backend error | re-throws ❌ | converges ✅ |
| `exception: 'ignore'` with real backend | no-op, crashes | swallows + converges |
| discriminator | prototype identity (`instanceof` own class) | structural (`ConstraintError` + `constructor.name`) |

### the "aha" moment

the value clicks when a user swaps the in-memory example cache for a real distributed backend and convergence *keeps working*. today it silently only works in unit tests (which use the local class). the fix makes the wrapper's promise — "converge on conflict" — hold across the package boundary where distributed races actually happen.

---

## user experience

- **usecase**: atomic conditional-write with graceful loser-convergence, backed by *any* conforming cache — including third-party backends.
- **contract inputs**: unchanged — `condition: { version, exception }` on `withSimpleCache` / `withSimpleCacheAsync`.
- **contract outputs**: unchanged — the loser returns the winner's value (or its own output as fallback if the winner vanished).
- **timeline**: no migration. the fix is internal to the converge guard; extant callers get correct behavior automatically once they use a cross-package backend.

there is **no new public surface**. this is a correctness repair behind the extant contract.

## mental model

- **how a user describes it to a friend**: "the cache wrapper's 'if I lose the race, just use the winner's value' feature now works with real cache backends, not just the built-in test one."
- **analogy**: two people reach for the last checkout slot. before, the loser only steps aside if they're literally the same person as the winner (never true across services). after, the loser steps aside whenever they recognize "that's a slot-already-taken signal" — regardless of which store issued it.
- **our terms vs theirs**: we say "converge on write conflict", "condition precondition", "structural discriminator". they say "handle the race", "put-if-absent", "don't crash when someone beats me to it".

## evaluation

### how well it solves the goal

fully. the wish names exactly one defect and one fix. option 1 (structural name check) repairs it with zero new public surface and zero work required of backend authors — any backend whose error is named `SimpleCacheConditionError` and extends `ConstraintError` is recognized.

### pros

- **cross-boundary correctness**: recognizes any conforming backend's error.
- **zero migration**: no contract change, no consumer action.
- **grounded in verified mechanics**: `HelpfulError` builds its prefix from `new.target.name`, and `error.constructor.name` is the runtime class name — both stable structural signals (verified below).
- **anchored in a documented contract**: the `SimpleCache` type already declares that a conditional backend "throws SimpleCacheConditionError on mismatch" and that "the package defines the vocabulary, each backend supplies the impl" (`SimpleCache.ts:148,150`). so the discriminator matches a name the contract already prescribes — not a hopeful guess.
- **consistent with a prior structural-detection precedent**: `HasCacheConditionals` already detects capability *structurally, not nominally* (`SimpleCache.ts:188-191`). a structural error-discriminator fits the grain of the codebase rather than a novel hack.
- **symmetric fix**: identical guard in sync + async wrappers, honoring `rule.require.wrapper-symmetry`.

### cons / tradeoffs

- **relies on the class name surviving**: safe for typical node libs; only at risk under aggressive minification that renames classes. acceptable for a server-side cache library.
- **structural, not nominal**: an unrelated `ConstraintError` subclass *also* named `SimpleCacheConditionError` would match. in practice the name is domain-specific enough that this is a non-risk.
- **stringly-typed discriminator**: `constructor.name` is a magic string; must be kept in lockstep with the class name. mitigated by deriving the expected name from the class itself (`SimpleCacheConditionError.name`) rather than a hardcoded literal.
- **⚠️ `instanceof ConstraintError` is itself package-fragile**: option 1 pairs the name check with `error instanceof ConstraintError`. but `instanceof` against *our* `ConstraintError` shares the exact prototype-identity weakness the fix exists to overcome — if a backend loads its own duplicate copy of `helpful-errors` (a real pnpm scenario), its `ConstraintError` is a different constructor and `instanceof ConstraintError` returns `false`, so option 1 would STILL fail to converge. the fix is only fully package-agnostic if we either (a) accept a single-copy-of-helpful-errors assumption, or (b) drop `instanceof` and walk the prototype-chain constructor names instead (assert some ancestor is named `ConstraintError`). this must be decided at the execution stone — see Q4.

### edge cases & pit-of-success

| edge case | how the contract keeps users safe |
|-----------|-----------------------------------|
| same-package error (extant tests) | still `instanceof ConstraintError` + correct name → still converges |
| a *different* ConstraintError (e.g. `BadRequestError` from the non-conditional-cache guard) | correct name check excludes it → still throws (loud, correct) |
| winner value vanished before re-read | extant fallback returns own output (unchanged behavior) |
| non-Error thrown from backend | `instanceof ConstraintError` is false → re-throws (correct) |
| backend throws a *differently-named* ConstraintError (non-conforming to the `SimpleCache.ts:148` contract) | name/marker check excludes it → still re-throws. the contract expects the name, but it is doc-enforced, not compile-enforced — a non-conforming backend breaks silently (see Q4/Q5) |
| eventually-consistent backend (loser re-reads before winner propagates) | *prior behavior, not introduced by this fix*: re-read may return `undefined` (→ own-output fallback) or a stale-but-present value (→ converge on stale). strongly-consistent backends (modern S3, dynamodb, redis) are unaffected. flagged so it is documented, not silently owned |

## open questions & assumptions

### assumptions made

1. **the fix belongs in this repo.** confirmed — this *is* the `with-simple-cache` repo; the converge guard lives at `withSimpleCache.ts:306` and `withSimpleCacheAsync.ts:357`. the wish's "no change to hand off from our repo" was written from the *consumer's* perspective; here we are the wrapper.
2. **option 1 (structural name check) is the chosen approach.** it is the cheapest and the wish lists it first. options 2/3 were not enumerated in the wish text — assume option 1 unless the wisher says otherwise.
3. **the expected class name should be derived, not hardcoded.** prefer `error.constructor?.name === SimpleCacheConditionError.name` over a string literal, to keep the discriminator honest if the class is ever renamed.

### open questions (triaged)

each question is tagged **[answered]** (settled now via logic/code), **[research]** (defer to research phase), or **[wisher]** (only the wisher can decide).

- **Q1 [answered — REVISED at execution]**: this vision leaned "derive the name from the class". the execution stone reversed it: a *derived* `SimpleCacheConditionError.name` would follow *our* rename and stop to match conformant foreign backends (whose class keeps the old name), and would mangle first under our own minification — so the literal is the correct cross-package wire string. it is extracted to a named const with the rationale inline. see the "design reconciled vs vision Q4" section of `5.1.execution.from_vision.yield.md` for the full rationale.
- **Q2 [answered]**: replace the `instanceof SimpleCacheConditionError` check outright rather than keep it as a fast-path — the structural/marker check is a strict superset, so a fast-path adds only a second code path to maintain. settled via logic.
- **Q3 [answered]**: yes — extract a shared internal predicate (`isSimpleCacheConditionError(error)`) so the sync and async guards cannot drift. settled via logic: satisfies `rule.require.wrapper-symmetry` via a single source of truth; there are exactly two guard sites (verified), so one predicate covers both.
- **Q4 [wisher]**: which discriminator lane? this is a genuine design decision because it shapes the *cross-package contract* backends must honor — the wisher owns that call. lanes, most to least robust:
  - **(recommended) branded marker** — give `SimpleCacheConditionError` a stable own signal (e.g. `static readonly isSimpleCacheConditionError = true` or a unique instance tag) and check `error?.isSimpleCacheConditionError === true`. immune to BOTH minification AND helpful-errors duplicate copies. cost: a one-line addition to the error class (scope grows to guard + `SimpleCacheConditionError.ts` — see Q5).
  - **(b) prototype-chain names** — assert an ancestor constructor is named `ConstraintError` and the class is named `SimpleCacheConditionError`. identity-free but still name/minification-fragile.
  - **(c) option 1 as-written** — `instanceof ConstraintError && constructor.name === 'SimpleCacheConditionError'`. simplest, but relies on a single shared `helpful-errors` copy AND class names that survive minification.
  - I lean **the branded marker**; it removes every identity/minification fragility for one extra line.
- **Q5 [answered]**: scope is *conditional on Q4* — the branded-marker lane touches `src/utils/errors/SimpleCacheConditionError.ts` (add the marker) plus the two guards; lanes (b)/(c) stay guard-only. settled via logic (a direct consequence of Q4's options).
- **Q6 [wisher]**: can the wisher name the concrete backend package that hit this cross-package re-throw? a real shape lets the test mirror reality (and could justify a real integration test alongside the synthetic unit test). only the wisher knows.
- **Q7 [wisher]**: the `SimpleCache.ts:148` contract that backends "throw SimpleCacheConditionError" is doc-only, not compile-enforced. should we strengthen that contract note (or add a documented conformance requirement) so a nonconformant backend is a known, documented gap rather than a silent break? tied to Q4; the wisher owns the contract's strictness.

### to research later

none — the **[research]** list is empty. the only external dependency (`helpful-errors`) is already verified in groundwork below; no open question requires external research.

## groundwork

### external research

- **helpful-errors `HelpfulError`** (`node_modules/.pnpm/helpful-errors@1.7.3/.../HelpfulError.js:27-36`): the message prefix is built from `new.target.name`, confirming both same- and cross-package `SimpleCacheConditionError` instances render `✋ SimpleCacheConditionError: …`. the class name is the runtime discriminator.
- **helpful-errors `ConstraintError`** (`.../ConstraintError.js:15-27`): extends `BadRequestError` → `HelpfulError` → `Error`; carries `code = { http: 400, exit: 2 }` and `emoji = '✋'`. `instanceof ConstraintError` is a stable cross-package base-class check (as long as both packages resolve the same `helpful-errors`, or the prototype chain name is used).
- constraint noted: structural name matching is minification-sensitive — acceptable for a node cache lib.

### internal research

- **the guard, sync** — `src/domain.operations/wrappers/withSimpleCache.ts:304-319`:
  ```ts
  if (!(conditionOption && error instanceof SimpleCacheConditionError)) throw error;
  ```
- **the guard, async** — `src/domain.operations/wrappers/withSimpleCacheAsync.ts:353-388`: identical shape (`instanceof SimpleCacheConditionError` at line 357).
- **the error class** — `src/utils/errors/SimpleCacheConditionError.ts:15`: `class SimpleCacheConditionError extends ConstraintError<{...}> {}` — none but a named `ConstraintError` subclass, exactly the structural shape the fix relies on.
- **public export** — `src/index.ts:37`: `SimpleCacheConditionError` is publicly exported, so backends *may* re-throw the exact class (same-package path) OR define their own (cross-package path). the fix must handle both.
- **extant tests** — `withSimpleCache.scope=conditionals.test.ts` and the async twin: every convergence test currently throws the *local* class, so they pass today and cannot detect the cross-package defect. the vision implies a **new test that simulates a cross-package error** (a `ConstraintError` subclass named `SimpleCacheConditionError` constructed from a *different* class object) to prove convergence works structurally.
- **symmetry rule** — `rule.require.wrapper-symmetry`: the fix must land identically in both wrappers.

### stdouts / vocab to reuse

- error message prefix `✋ SimpleCacheConditionError: …` (from `HelpfulError` + `ConstraintError.emoji`).
- domain terms already in the code: `conditionOption`, `convergeOnWriteConflict`, `condition.version`, `exception: 'ignore' | 'throw'`.

## what is awkward?

- **stringly-typed discriminator**: matching on `constructor.name` is inherently a magic-string check. it fights the "shapefit / no magic values" instinct. mitigation: derive the name from the class (`SimpleCacheConditionError.name`) and centralize it in one predicate so the string lives in exactly one place.
- **two truths for "is this our conflict?"**: `instanceof` (nominal) vs `constructor.name` (structural). the structural check is strictly more permissive; keeping both would be redundant. the awkwardness is choosing to *loosen* an identity check — it feels like weakening type safety, but it is the only signal that survives package boundaries.
- **the wish framing**: the wish reads as a hand-off *to* with-simple-cache from a consumer repo, but we *are* with-simple-cache. slight cognitive dissonance settled by reframing: "we are the wrapper the peer was pointing at."
- **minification caveat**: reliance on class names is philosophically uncomfortable for a library that could theoretically be bundled. acceptable here, but worth an inline comment so future readers know the tradeoff was deliberate.
