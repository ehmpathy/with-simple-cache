## CRITICAL: type contract matrix (BLOCKING)

**the type system IS the feature.** the compiler must prevent misuse at definition time.

### the matrix: `meta: 'include'` x cache declaration x has uri

this is the ONLY important part. 8 cells, 8 type tests, both wrappers symmetric.

| cache declaration | has `uri` | `meta: 'include'` | expected |
|-------------------|-----------|-------------------|----------|
| static | yes | allowed | compiles |
| static | no | forbidden | TYPE ERROR |
| dynamic extraction | yes (`WithCacheUri`) | allowed | compiles |
| dynamic extraction | no (`SimpleCache`) | forbidden | TYPE ERROR |

**x2 wrappers = 8 cells total**

### why dynamic extraction is the critical case

```ts
// THE USE CASE: cache comes from context, type must enforce WithCacheUri
const _runPlay = async (
  input: { query: string },
  context: { cache: WithCacheUri<SimpleCache<string>> },  // <-- MUST require WithCacheUri
): Promise<string> => fetchData(input.query);

const runPlay = withSimpleCacheAsync(_runPlay, {
  cache: ({ fromInput }) => fromInput[1].cache,
  meta: 'include',  // <-- compiles BECAUSE context.cache has uri
});

// if context.cache is SimpleCache (no uri), meta: 'include' MUST NOT compile
```

### acceptance criteria

| criterion | mechanism | verification |
|-----------|-----------|--------------|
| static + uri → compiles | `HasCacheUri<C>` detects uri | `@ts-expect-error` absent |
| static + no uri → TYPE ERROR | `MetaValue<C>` excludes `'include'` | `@ts-expect-error` present |
| dynamic + `WithCacheUri` → compiles | extraction return type propagates | `@ts-expect-error` absent |
| dynamic + `SimpleCache` → TYPE ERROR | extraction return type propagates | `@ts-expect-error` present |
| symmetric across wrappers | same types in both | both type test files |

**the feature is incomplete until all 8 type tests pass.**

## the outcome world

### before

```ts
// wrap the raw function with cache (cache extracted from context arg)
const runPlayToDownloadCsv = withSimpleCacheAsync(
  _runPlayToDownloadCsv,
  { cache: ({ fromInput }) => fromInput[1].accela.artifact.cache.instance },
);

const csv = await runPlayToDownloadCsv(input, context);

// caller must manually reconstruct cache uri after the fact
// manual uri reconstruction — error-prone, duplicates knowledge
const cacheKey = JSON.stringify(input);
const cacheUri = `s3://${context.accela.artifact.cache.bucket}/${cacheKey}`;
```

problems:
- caller must know cache internals (bucket name, key format)
- uri reconstruction duplicates knowledge from cache config
- no type safety — typos in bucket/prefix go unnoticed
- pattern repeated everywhere cache output location is needed

### after

```ts
// wrap the raw function with cache + include metadata (cache extracted from context arg)
const runPlayToDownloadCsv = withSimpleCacheAsync(
  _runPlayToDownloadCsv,
  {
    cache: ({ fromInput }) => fromInput[1].accela.artifact.cache.instance,
    meta: 'include',
  },
);

const { output: csv, cached } = await runPlayToDownloadCsv(input, context);
// cached.uri available when cache supports it (e.g., s3)
const cacheUri = cached.uri; // 's3://bucket/prefix/key'
```

benefits:
- cache uri comes from the source of truth (the cache itself)
- no manual reconstruction
- type-safe — `cached.uri` only available when cache supports it
- single location for uri logic

### the "aha" moment

"wait, the cache already knows where it stored the data — why am i reconstructing the uri myself?"

the value clicks when developers realize they no longer need to:
1. import the bucket name from config
2. manually concatenate `s3://${bucket}/${key}`
3. hope they got the prefix right
4. maintain two sources of truth (cache config + uri reconstruction)

## user experience

### usecases

| usecase | goal |
|---------|------|
| cache artifact reference | return uri to caller for downstream use |
| cache artifact logging | log where data was persisted |
| cache artifact linking | store uri in database for later retrieval |
| cache artifact export | expose uri in api response |

### contract inputs

```ts
interface WithSimpleCacheAsyncOptions<L, C> {
  cache: /* ... extant ... */;
  serialize?: /* ... extant ... */;
  deserialize?: /* ... extant ... */;
  expiration?: /* ... extant ... */;
  bypass?: /* ... extant ... */;

  // new
  meta?: 'include' | 'exclude';  // default: 'exclude'
}
```

### contract outputs

when `meta: 'exclude'` (default, backwards compat):
```ts
const cachedFn: L = withSimpleCacheAsync(logic, { cache, meta: 'exclude' });
const result = await cachedFn(...args);
```

when `meta: 'include'`:
```ts
const cachedFn = withSimpleCacheAsync(logic, { cache, meta: 'include' });
const { output, cached } = await cachedFn(...args);
// cached.uri?: string — present when cache supports uri exposure
```

**why `cached.uri` but not `cached.key`?**

the key is derived from the function arguments via serialization — the caller already has the inputs. returning the derived key would add little value.

the uri is different: the **cache** knows the full path (bucket + prefix + key), but the **caller** only knows the inputs they provided. without `cached.uri`, the caller must replicate key derivation logic and then reconstruct the uri from scattered config — exactly the problem this feature solves.

### timeline

1. developer adds `meta: 'include'` to extant call
2. typescript narrows return type to `{ output, cached }`
3. developer accesses `cached.uri` if cache supports it
4. manual uri reconstruction code deleted

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| eliminate manual uri reconstruction | yes | cache exposes its own uri |
| type safety | yes | `cached.uri` only present when supported |
| backwards compat | yes | `meta: 'exclude'` is default |
| zero runtime cost for non-users | yes | no change to default path |

### pros

- single source of truth for uri
- type-safe access to cache metadata
- composable — `cached` object can grow with more metadata
- follows extant pattern from `withExtendableCacheAsync`

### cons

- adds complexity to return type (conditional on meta option)
- `cached.uri` availability depends on cache implementation

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| in-memory cache (no uri) | `cached.uri` is `undefined` or absent |
| custom cache without uri | same — `cached.uri` absent |
| s3 cache | `cached.uri` is `s3://bucket/prefix/key` |
| on-disk cache | `cached.uri` could be `file:///path/to/file` |

pit of success:
- typescript prevents access to `cached.uri` with `meta: 'exclude'`
- typescript narrows `cached.uri` type based on cache implementation (if possible)

## what is awkward?

### meta option name

**decision:** `meta: 'include' | 'exclude'`

**return shape:** `{ output, cached: { uri? } }`

**why `meta` over `output`:**

| aspect | `output: 'nested' \| 'direct'` | `meta: 'include' \| 'exclude'` |
|--------|-------------------------------|-------------------------------|
| describes | structure | content |
| communicates benefit | no (what, not why) | yes (you get metadata) |
| collision risk | slight (`cache.output`) | none |
| values | abstract | action verbs |

`meta: 'include'` directly says "include metadata" — the `cached` property in the return is that metadata.

**alternatives considered:**

| option | values | rejected because |
|--------|--------|------------------|
| `mode` | `'nested'` | overloaded — could mean execution mode |
| `output` | `'nested' \| 'direct'` | describes structure not content; slight collision with `cache.output` |
| `transform` | `'nest' \| 'pass'` | collides with serialize/deserialize concepts |
| `nest` | `true \| false` | boolean limits future extensibility |

**why `cached` not `cache` in return:**

the option `cache` already exists (the cache instance). returning `{ output, cache }` would shadow it in destructure scope. `cached` is the metadata about what was cached — past tense indicates it's info about the cache operation, not the cache itself.

**recommendation:** use `meta: 'include' | 'exclude'` with return shape `{ output, cached }`

### cached.uri availability

`cached.uri` is only meaningful for caches that persist to addressable locations (s3, disk). in-memory caches have no uri. this creates an optional property that may confuse users.

mitigation: typescript narrowing based on cache type, or explicit `undefined` for in-memory.

### extending SimpleCache interface

adding `uri` to `SimpleCache` interface would require changes to all cache implementations. alternative: wrapper that decorates cache with uri capability.

recommendation: use a decorator pattern or optional `uri` getter on cache, not a required interface change.

### backwards compatibility

`meta` option is backwards compatible (default is `exclude`). the enum `'include' | 'exclude'` is focused and unlikely to grow — it's a binary choice about whether to include metadata.
