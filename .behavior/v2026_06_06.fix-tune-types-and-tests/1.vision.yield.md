# vision: fix-tune-types-and-tests

## the outcome world

### before

```typescript
// developer types "Simple..." in IDE
SimpleSyncCache     // appears first (alphabetically)
SimpleAsyncCache    // appears second
SimpleCache         // appears third

// mental friction: "is it Simple-Sync or Sync-Simple?"
// autocomplete doesn't group by noun prefix
```

tests scattered — some in main files, some in scoped files, no clear pattern:
```
withSimpleCache.test.ts              # 378 lines, mixed concerns
withSimpleCacheAsync.test.ts         # 529 lines, mixed concerns
withSimpleCache.scope=serde.test.ts  # scoped
withSimpleCache.scope=meta.test.ts   # scoped
...
```

### after

```typescript
// developer types "SimpleCache..." in IDE
SimpleCache        // base union type
SimpleCacheAsync   // async variant
SimpleCacheSync    // sync variant

// noun-first: all cache types grouped
// autocomplete shows all variants as you type "SimpleCache"
```

tests organized by concern — main files thin, all behavior in scoped files:
```
# withSimpleCache (sync wrapper)
withSimpleCache.test.ts                              # thin: base happy path
withSimpleCache.scope=sync-logic-sync-cache.test.ts  # sync logic, sync cache
withSimpleCache.scope=async-logic-sync-cache.test.ts # async logic, sync cache
withSimpleCache.scope=invalidation.test.ts
withSimpleCache.scope=serde.test.ts                  # already exists
...

# withSimpleCacheAsync (async wrapper)
withSimpleCacheAsync.test.ts                              # thin: base happy path
withSimpleCacheAsync.scope=async-logic-sync-cache.test.ts # async logic, sync cache
withSimpleCacheAsync.scope=async-logic-async-cache.test.ts # async logic, async cache
withSimpleCacheAsync.scope=invalidation.test.ts
withSimpleCacheAsync.scope=serde.test.ts                  # already exists
...
```

### the "aha" moment

developer opens autocomplete, types `SimpleCache`, sees all variants. no more "is it `SimpleSyncCache` or `SyncSimpleCache`?" — the name just works.

developer opens test directory, sees each concern isolated. wants to add an invalidation test? `scope=invalidation.test.ts`. wants to fix a serde bug? `scope=serde.test.ts`. no more 500-line file traversal.

## user experience

### use case 1: type a cache interface

**goal:** declare a cache type constraint

**before:**
```typescript
import { SimpleSyncCache } from 'with-simple-cache';
//         ^ must remember "Sync" comes second
```

**after:**
```typescript
import { SimpleCacheSync } from 'with-simple-cache';
//         ^ type "SimpleCache" → autocomplete shows all variants
```

### use case 2: find tests for a specific behavior

**goal:** add or fix tests for cache invalidation

**before:**
1. open `withSimpleCacheAsync.test.ts`
2. ctrl+f "invalidation"
3. scroll through 529 lines to find the describe block
4. edit among unrelated tests

**after:**
1. open `withSimpleCacheAsync.scope=invalidation.test.ts`
2. all invalidation tests in one place
3. edit in focused context

### use case 3: understand test coverage

**goal:** verify a feature is tested

**before:** grep through multiple large files, unclear what's where

**after:** file names tell you: `scope=bypass`, `scope=meta`, `scope=async-logic-async-cache`

## mental model

### how users describe this to a friend

> "all the cache types start with `SimpleCache` now, so autocomplete groups them"

> "each test file is named for what it tests — invalidation tests in the invalidation file"

### analogies

**type rename:** like `ColorRed`, `ColorBlue` instead of `RedColor`, `BlueColor` — group by noun, qualify with adjective

**test scopes:** like unix philosophy — each file does one task well

### terms

| we say | users might say |
|--------|-----------------|
| noun-adjective order | "grouped by prefix" |
| scoped test files | "organized tests" |
| `SimpleCacheAsync` | "async cache type" |

## evaluation

### how well does it solve the goals?

| goal | solution | fit |
|------|----------|-----|
| autocomplete groups cache types | noun-first names | excellent |
| tests easy to navigate | scoped files | excellent |
| tests easy to extend | isolated concerns | excellent |

### pros

- autocomplete discovery improves
- test files are small and focused
- follows established repo patterns (`rule.require.order.noun_adj`)
- symmetric with extant `scope=*` test files

### cons

- compat-break for type imports (minor — just rename)
- more test files in directory (mitigated by clear names)
- migration effort (one-time)

### edge cases

| edge case | pit of success |
|-----------|----------------|
| user imports old type name | typescript error at compile time — clear signal |
| tests reference wrong scope | file name describes content — easy to find |
